# Clarify — feature / 需求不明確時追問

訊息是「要求 / 委託 / feature request」但細節不清楚時，**回追問而不是回承諾或瞎猜實作**。

## 觸發條件

滿足以下任一就判定「需求不明確」：

- 動詞是「加 / 做 / 改 / 弄 / 新增 / 支援 / 處理 / 幫忙 + 名詞」，但對象或 scope 模糊
  - 例：「幫我加個 export 功能」、「那個 bug 修一下」、「搞一個 API」
- 指涉詞不清（「那個」、「這個」、「XX 那邊」）且 thread context 也還原不出來
- 沒講清楚「改哪裡 / 給誰用 / 什麼時候要 / 什麼叫做完」中任何一個關鍵資訊
- 對方只丟連結或截圖但沒說要幹嘛

## 追問原則

### 1. 最多問 1-2 題，別連珠砲
chat 不是 PM 需求訪談。一次問 1 題最理想，最多 2 題；對方答完再問下一輪。

### 2. 問「最阻塞」的那一題
優先序（缺就先問）：
1. **scope / what** — 到底要改什麼？（功能 A 還 B？哪個 page / API？）
2. **where** — 哪個 repo / 服務 / 檔？（多 repo 環境下很常見）
3. **who / use case** — 誰會用？什麼情境？（決定設計方向）
4. **when / priority** — 要多急？（決定要不要先放著）
5. **done criteria** — 怎樣算做完？（測試？上 prod？）

### 3. 用封閉式問題比開放式好
- 好：「你是指 A 還是 B？」、「前端改還後端改？」、「今天要還週末前？」
- 不好：「可以多說明一下嗎？」、「你想怎麼做？」— 太空洞，對方也不想打字

### 4. 不要在追問時「順便」給建議
- 壞：「你是要 A 還 B？我覺得應該 A 會比較好因為 XXX」← 對方還沒決定你就倒意見
- 好：「你要 A 還 B？」— 決定了再出意見

### 5. 追問語氣貼 style profile
保持口語短句：
- 「蛤你是說 XXX 嗎」
- 「哪個 repo？」
- 「要改前端還後端」
- 「今天要還晚點都可以」
- 「是 YYY 那個 bug 嗎？」

## 典型範例

### 範例 1：模糊 feature request
```
對方：「幫我加個 export 功能」
壞回：「好，我看一下」← 什麼都沒確認就接單
壞回：「好，我做個 CSV export 加 filter 和分頁」← 瞎猜 scope
好回：「export 哪個列表？CSV 還 Excel？」
```

### 範例 2：指涉不清
```
對方：「那個東西可以改一下嗎」
好回：「你說哪個？」← 先還原指涉，再談細節
```

### 範例 3：只丟連結
```
對方：「[Jira ticket URL]」（沒說要幹嘛）
步驟：先 fetch ticket（jira.md）→ 看 assignee 是不是自己
若是自己 → 「看到了，你要我先看這張？」
若不是 → 「這張不是我的耶，你要問 @assignee」
```

### 範例 4：多選項可能
```
對方：「chat-bot 那邊可以跟 Jira 整合嗎」
好回：「你是指讓 bot 回覆的時候自動抓 ticket，還是想要另外做同步？」
```

### 範例 5：急迫性不明
```
對方：「這個 feature 要做」
好回：「今天要還這週內？」← 先抓時程決定要不要插隊
```

## 何時**不**追問

- 對方問的是**事實**（「XX 在哪」、「這個能不能用」）→ 直接查，別倒問回去
- 追問第二輪仍不清楚 → 回「我看一下再跟你 sync」+ skip，讓使用者自己接手
- 對方已經提供足夠 context → 別為了問而問
- 紅線（金錢 / 合約 / 對外）→ 永遠 skip，不要追問（追問本身就是介入）

## 追問 vs fallback 的分界

| 狀態 | 做法 |
|------|------|
| 完全沒 context，連猜都不能猜 | 追問 |
| 有方向但關鍵細節缺 | 追問 |
| 細節還好，只是怕瞎做 | 直接做 / 查，回「我先看看 XX，不對再問你」 |
| 追問過一輪還是沒回 | fallback「我先放著，你 ready 再 ping」+ skip |
