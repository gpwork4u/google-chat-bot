## 1. 背景

Sprint 5 收尾後（CR-001 D-skip 上線、loop 真正 incremental），下一個瓶頸是**回覆品質**：

- 每個 space 都有自己的「產品 / 角色 / 術語 / 共識 / 人物關係」，skill 目前只看 anchor + thread + around context（短期記憶），**長期記憶完全沒有**
- 例：某 space 是「fedflow-team」討論 K8s controller、另一個 space 是「家庭 chat」，skill 不知道兩者在講什麼產品、自己（local user）在這個 space 是什麼角色（owner / 旁聽 / 客戶 / 親人）
- profile facts (`user_profile_facts`, F-004) 是**全域**個人資訊（我家在哪 / 我寵物名字），不是 per-space 語境
- 結果：skill 回覆常常太通用、術語對不上、語氣偏掉

同時實機運作 v0.4.0 + CR-001 上線後發現第二個小痛點：

- backend pending list 純粹是 API（`GET /api/claude/pending`），使用者無法用 UI 看「現在 backend 認為哪些訊息是 pending」
- Approval queue (F-002) 只看「skill 已產 draft」的訊息，**沒產 draft 也沒 skip 的中間狀態看不到**
- Debug 跟手動 skip / unskip 都要敲 curl

**CR-002 同時解這兩個問題**（合併進一個 CR 因為共用 backend message reading layer + 同 sprint scope）：

- **Part A（Sprint 7）**：scrape 全歷史 → LLM 萃取 → UI approve → 存 `space_facts` → skill 注入 prompt
- **Part B（Sprint 6）**：Pending message viewer UI 頁面，含篩選 + 手動 skip / unskip

## 2. 變更概要

### 2.1 新增

#### Part A — Space facts pipeline（Sprint 7）

- **Chrome extension**：popup 加「Sync history for this space」按鈕 + popup 主畫面加「Sync all spaces」按鈕。透過 batchexecute 拉**全 space 歷史**（不是 incremental — 拿整段 thread / topic 列表），寫進 backend
- **Backend**：
  - 新 endpoint `POST /api/extension/sync-history` 接 extension 推來的歷史訊息（idempotent — 用 message_id PK 去重）
  - 新 endpoint `POST /api/extension/sync-history/start`（開始 job）+ `GET /api/extension/sync-history/status?job_id=...`（查進度）— 因為一個 space 可能幾千則訊息，要分批
  - 新 table `space_history_sync_jobs` 追蹤進度（per-space）
  - 新 table `space_facts`（仿 user_profile_facts 加 space_key 與 source_message_ids[]）
  - 新 endpoints：`GET/POST/PATCH/DELETE /api/space-facts` + `GET /api/space-facts/candidates` (mining candidates queue)
- **Skill**：`.claude/skills/space-facts-mining/`（新 skill；類似 chat-drafts 結構）
  - Loop: 讀 `space_facts_mining_jobs` queue → 對每個 space 拉最近未處理訊息 → LLM 列點「產品 / 角色 / 術語 / pinned 決策 / 人物關係」5 類 → POST 進 `space_facts` 表（status=`candidate`）
- **Frontend** SettingsPage 加「Space facts」section：
  - 列已 approved facts (per-space 分組)
  - 「待審核 mining candidates」入口 → 連到新頁 `/space-facts/candidates`
  - 新頁 `/space-facts/candidates`：每條 candidate 顯示 sentence + visibility + source messages（可展開看原文）+ 三個動作（approve / edit / reject）
- **chat-drafts skill 整合**：在 step 1（拉 pending）後、step 2（拉 context）前，依 anchor `space_key` 拉 `GET /api/space-facts?space_key=...` 注入 prompt context

#### Part B — Pending message viewer（Sprint 6）

- **Frontend**：新頁 `/pending`：
  - 顯示 backend 過濾後的 pending message 列表（沒 draft + 沒 skip）
  - 欄位：message_id / space_name / sender_name / body (truncated 100 chars，hover/expand 看全文) / observed_at / mentioned / 「為何 pending」說明
  - 篩選工具列：space (multi-select dropdown) / sender (free text) / contains keyword (search input) / mentioned only checkbox
  - 每筆 row 有「Skip」按鈕（彈 reason 選單：pure-ack / overheard / policy-redline / not-targeted / low-info / manual-other）→ POST `/api/claude/skip`
  - 已 skip 訊息頁籤切換（Tab：Pending / Skipped）— skipped 頁顯示 unskip 按鈕
- **Backend**：
  - `GET /api/claude/pending` 加 query params 支援 UI 篩選：`space_key`, `sender_contains`, `body_contains`, `mentioned_only`, `limit`, `offset`
  - `GET /api/claude/skipped` 加同樣的 query params + 已存在的 `since` / `by`
- **WS event**：新增 `pending_changed` event（任何 pending 變動 — 新訊息進、被 skip、被處理成 draft — 都廣播），讓 UI 即時更新

## 3. 候選方案決策（重要決策快速記錄）

### 3.1 全歷史同步觸發方式

| 選項 | 結論 |
|------|------|
| A：Chrome extension popup 手動按鈕 | ✅ **採用** — user 決定何時跑，不消耗 Google 端 quota 在背景。批次/per-space 兩種按鈕。 |
| B：backend 定時排程 batchexecute | ❌ 拒絕 — backend 沒 session cookie，batchexecute 必須在 extension MAIN-world 跑 |
| C：頁面開啟自動同步 | ❌ 拒絕 — 噪音太大，可能跑很久 |

### 3.2 LLM mining 執行位置

| 選項 | 結論 |
|------|------|
| A：backend 內建 Claude API call | ❌ 拒絕 — 跟現有架構不一致（沒人在 Go 端打 Claude），key 管理 / 成本控制麻煩 |
| B：新 skill `space-facts-mining` 走 Claude Code loop | ✅ **採用** — 跟 chat-drafts 同模式，user 用 `/loop` 跑，把 mining queue 拉乾淨 |
| C：手動觸發每次只 mine 一個 space | ⏸ 折衷可選 — Skill 設計上同時支援「給定 space_key 只跑該 space」與「全 queue」兩種模式 |

### 3.3 Space facts 資料模型

| 選項 | 結論 |
|------|------|
| A：仿 user_profile_facts 加 `space_key` + `source_message_ids` | ✅ **採用** — 已驗證的 schema 模板，加 visibility public/private/secret + status (candidate/approved/rejected) |
| B：純自由文字單欄位 markdown | ❌ — UI approve 要逐筆操作，必須拆 row |
| C：5 個分類 table（products / roles / glossary / decisions / relations） | ❌ — 過度設計，未來要加類別還要 migrate |

**採 A**：用 `category` 欄位 enum 區分 5 類（product / my-role / glossary / pinned-decision / relation），結構彈性、UI 可分組顯示。

### 3.4 Facts 注入 prompt 的時機

| 選項 | 結論 |
|------|------|
| A：在 chat-drafts step 1 後立即拉並注入 system prompt | ✅ **採用** — 早注入避免 step 3 分類 / step 4 起草都用得到 |
| B：起草階段才拉 | ❌ — 分類時就需要術語 / 角色判斷 |
| C：cache per session（一輪 loop 內第一次拉、後續重用） | ✅ **同時採用** — 跟 style-profile / profile cache 同策略 |

### 3.5 Pending viewer 與 Approval queue 關係

| 選項 | 結論 |
|------|------|
| A：兩個分開頁，pending 是「raw queue」，approvals 是「skill 加工後」 | ✅ **採用** |
| B：合併同一頁 + tabs | ❌ — 不同心智模型（pending 是 debug / 巡邏視角；approval 是動作視角） |

### 3.6 全 space sync 進度回報

| 選項 | 結論 |
|------|------|
| A：sync 過程同步阻塞 popup（spinner 直到完成） | ❌ — 一個 space 幾千則訊息可能跑數分鐘 |
| B：背景跑、popup 顯示「同步中… N / M spaces」+ poll status endpoint | ✅ **採用** |
| C：WS push 進度 | ⏸ 候選優化 — Sprint 7 若時間夠就升級用 WS（Sprint 6 先用 polling） |

### 3.7 Mining 觸發頻率

預設值（在 CR 中記，**TBD 等實機運作再 tune**）：
- 每個 space 至少累積 50 則新訊息才進 mining queue
- 同一 space 7 天內不重 mine（避免重複生 candidate）
- Mining skill 一輪最多處理 3 個 space（避免 token 爆掉）

## 4. 詳細設計

### 4.1 Migration 0020: space_facts

```sql
-- internal/store/migrations/0020_space_facts.sql

CREATE TABLE IF NOT EXISTS space_facts (
    id BIGSERIAL PRIMARY KEY,
    space_key TEXT NOT NULL,
    category TEXT NOT NULL CHECK (category IN (
        'product', 'my-role', 'glossary', 'pinned-decision', 'relation'
    )),
    content TEXT NOT NULL,
    visibility TEXT NOT NULL DEFAULT 'private' CHECK (visibility IN (
        'public', 'private', 'secret'
    )),
    status TEXT NOT NULL DEFAULT 'candidate' CHECK (status IN (
        'candidate', 'approved', 'rejected'
    )),
    source_message_ids BIGINT[] NOT NULL DEFAULT '{}',
    note TEXT NOT NULL DEFAULT '',
    created_by TEXT NOT NULL CHECK (created_by IN (
        'mining-skill', 'manual'
    )),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    approved_at TIMESTAMPTZ NULL
);

CREATE INDEX idx_space_facts_active
    ON space_facts (space_key, category)
    WHERE status = 'approved';

CREATE INDEX idx_space_facts_candidates
    ON space_facts (space_key, created_at DESC)
    WHERE status = 'candidate';

COMMENT ON COLUMN space_facts.content IS
  '單條 markdown sentence，例：「此 space 主要討論 fedflow K8s controller，重點在 reconciler queue」';
COMMENT ON COLUMN space_facts.source_message_ids IS
  '此 fact 從哪幾則 messages.id mining 出來（追溯來源）';
```

### 4.2 Migration 0021: space_history_sync_jobs

```sql
-- internal/store/migrations/0021_space_history_sync_jobs.sql

CREATE TABLE IF NOT EXISTS space_history_sync_jobs (
    id BIGSERIAL PRIMARY KEY,
    job_id TEXT NOT NULL UNIQUE,        -- UUID, extension 端產生
    space_key TEXT NULL,                -- NULL = 全 space 同步
    status TEXT NOT NULL DEFAULT 'running' CHECK (status IN (
        'running', 'completed', 'failed', 'cancelled'
    )),
    total_messages INTEGER NOT NULL DEFAULT 0,
    inserted_messages INTEGER NOT NULL DEFAULT 0,  -- 真的 insert 進來的（去重後）
    duplicate_messages INTEGER NOT NULL DEFAULT 0,
    failed_messages INTEGER NOT NULL DEFAULT 0,
    started_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at TIMESTAMPTZ NULL,
    error_message TEXT NULL
);

CREATE INDEX idx_sync_jobs_recent ON space_history_sync_jobs (started_at DESC);
```

### 4.3 Migration 0022: space_facts_mining_jobs（queue）

```sql
-- internal/store/migrations/0022_space_facts_mining_jobs.sql

CREATE TABLE IF NOT EXISTS space_facts_mining_jobs (
    id BIGSERIAL PRIMARY KEY,
    space_key TEXT NOT NULL UNIQUE,    -- 一個 space 同時只能有一筆 active job
    status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN (
        'pending', 'running', 'completed', 'failed'
    )),
    last_mined_message_id BIGINT NULL,  -- 上次 mine 到哪筆 message
    last_mined_at TIMESTAMPTZ NULL,
    candidates_generated INTEGER NOT NULL DEFAULT 0,
    error_message TEXT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_mining_jobs_pending
    ON space_facts_mining_jobs (created_at DESC)
    WHERE status = 'pending';
```

### 4.4 API Contract

#### Part A endpoints

##### `POST /api/extension/sync-history/start`
Auth: localhost only。

Request:
```json
{ "job_id": "uuid-v4-string", "space_key": "spaces/AAA" }
```
`space_key` 可省略，表示全 space。

Response 201:
```json
{ "job_id": "uuid-v4-string", "status": "running", "started_at": "ISO 8601" }
```

##### `POST /api/extension/sync-history`
Auth: localhost only。Extension push 批次訊息上來。

Request:
```json
{
  "job_id": "uuid-v4-string",
  "messages": [
    {
      "message_id": "spaces/AAA/messages/BBB",
      "space_key": "spaces/AAA",
      "space_name": "Team #frontend",
      "thread_key": "TP123",
      "sender_id": "users/CCC",
      "sender_name": "Alice",
      "body": "...",
      "observed_at": "ISO 8601",
      "mentioned": false
    }
  ]
}
```

Response 200:
```json
{ "inserted": 42, "duplicates": 8, "failed": 0 }
```

Errors:
| Status | Code | Condition |
|--------|------|-----------|
| 400 | INVALID_INPUT | messages 為空 / 任一訊息缺必要欄位 |
| 404 | JOB_NOT_FOUND | job_id 不存在或已 completed/cancelled |

##### `GET /api/extension/sync-history/status?job_id=...`

Response 200: 整個 `space_history_sync_jobs` row。

##### `GET /api/space-facts`
Query: `space_key` (optional), `category` (optional), `status` (default `approved`), `visibility` (default 排除 secret)

Response:
```json
{
  "facts": [
    {
      "id": 1,
      "space_key": "spaces/AAA",
      "category": "product",
      "content": "此 space 主要討論 fedflow K8s controller",
      "visibility": "private",
      "status": "approved",
      "source_message_ids": [123, 124, 130],
      "note": "",
      "created_by": "mining-skill",
      "created_at": "...",
      "approved_at": "..."
    }
  ]
}
```

##### `GET /api/space-facts/candidates`
Query: `space_key` (optional), `limit` (default 50)

Response: 同 `GET /api/space-facts` 但 status=candidate。

##### `POST /api/space-facts`
Body: `{ space_key, category, content, visibility?, source_message_ids?, note?, created_by? }`

Response 201: 整個 row。`status` 預設依 `created_by`：mining-skill → `candidate`；manual → `approved`。

##### `PATCH /api/space-facts/{id}`
Body (partial): `{ content?, visibility?, status?, note? }`

當 `status` 從 `candidate` → `approved` 時 set `approved_at = NOW()`。

##### `DELETE /api/space-facts/{id}`

Response 200。

##### `POST /api/space-facts/{id}/approve`
便利 endpoint，等同 PATCH `{status: "approved"}`。

##### `POST /api/space-facts/{id}/reject`
便利 endpoint，等同 PATCH `{status: "rejected"}`。

##### `POST /api/space-facts/mining-queue`
Body: `{ space_key }` — 把該 space 加入 mining queue（手動觸發；不存在則 insert，存在則 reset status=pending）

#### Part B endpoints

##### `GET /api/claude/pending`（既有 + 新 query params）
新增 query params：
- `space_key` (optional)
- `sender_contains` (optional, case-insensitive substring on sender_name)
- `body_contains` (optional, case-insensitive substring on body)
- `mentioned_only` (`true` / `false`, default false)
- `limit` (default 50, max 200)
- `offset` (default 0)

Response 既有結構不變。

##### `GET /api/claude/skipped`（既有 + 新 query params）
同上 + 已存在的 `since` / `by`。

#### WS event

`pending_changed` — 任何 pending mutate 都廣播：
```json
{ "type": "pending_changed", "reason": "new_message" | "skipped" | "drafted" | "unskipped" }
```
Frontend 收到後 SWR revalidate pending list。

### 5.4 回歸風險

| 風險 | 對策 |
|------|------|
| Sync history 拉幾千則訊息可能爆 backend disk / DB | 批次插入 + 寫入前檢查 max-job-bytes（10MB / batch）；訊息只存必要欄位（不含 raw_events） |
| Mining skill 跑出來的 candidate 完全沒用 / 太空泛 | 第一輪在 fedflow-team 或某幾個 high-traffic space 試點，user approve rate < 50% 則 prompt 調整；CR 中先 ship，調 prompt 在 sprint 7 後段 |
| Pending viewer 大量訊息 render 卡頓 | 分頁（offset/limit），預設 limit=50，使用者主動翻頁；row 用 React.memo |
| WS event `pending_changed` 太頻繁 | Backend debounce 200ms |
| Manual skip 後使用者後悔（誤點） | Unskip 按鈕 + 5 秒 toast 含「復原」連結 |
| Sync 中斷 / extension popup 關閉 | Job 設 60 分鐘 timeout，超過標 failed；user 可重新觸發（idempotent — message_id PK 去重） |
| Facts 注入後 chat-drafts prompt 變太長 → token 超標 | 設 per-space facts 上限（每類最多 5 條），超過提示 user 整併；prompt 中只列前 5 條 |
| 不同 space 的 fact 互相污染 | facts 嚴格按 space_key 隔離，不跨 space 注入 |
